# dev-delivery

## 核心原则

1. **真实输入输出** — 所有判断基于实际文件、代码、命令输出，不基于猜测
2. **最小颗粒度** — 每个工作单元可独立开发、测试、验证
3. **测试先行或测试并行** — 不测完不认为开发完成，避免集成阶段大量返工
4. **集成点强制回归** — 拼在一起必须做回归测试，不能只说自己"测过了"
5. **交付中断 = 决策清单** — 卡住时列出所有待决策项，用户选择后继续
6. **可设里程碑** — 大项目分段交付，每通过一个里程碑再继续下一个

---

## 触发条件

收到项目需求、功能开发、bug 修复、或"帮我开发 X"类请求时加载此 skill。

---

## 流程总览

```
阶段 0：理解项目（探索）
阶段 1：编写开发测试规划
阶段 2：执行开发 + 测试
阶段 3：集成点回归测试
阶段 4：交付 + 决策清单（如有）
        ↓（如有决策且用户已决策）
阶段 2 ← 继续
```

---

## 阶段 0：理解项目

### 做什么
- 读项目 README、现有代码结构、配置文件
- 了解技术栈、入口文件、模块划分
- 确认是否有现有测试框架
- 识别项目规范（代码风格、测试框架、CI 配置）

### 产出
- 项目结构摘要（不超过 10 行）
- 技术栈清单
- 模块划分图（文字版即可）
- **已知约束和风险**（一开始就列出，不藏着）

### 原则
- 读真实的文件和代码，不用"通常来说"推测
- 如果代码库太大，用 glob/grep 抽样读，不要求全部读完

---

## 阶段 1：编写开发测试规划

### 规划格式（输出到文件）

```
# [项目名] 开发测试规划

## 目标
（一句话说清楚要做什么）

## 颗粒度拆分

### 模块 A
**做什么**：...
**依赖**：模块 B（必须先测通）
**开发工作**：
1. [工作内容] — 预期产出：...
2. [工作内容] — 预期产出：...

**测试计划**：
- [ ] 测试点 1：怎么做、预期什么结果
- [ ] 测试点 2：...

### 模块 B
（同上格式）

## 集成点回归计划

| 集成时机 | 做什么 | 验收标准 |
|---------|--------|---------|
| 模块 A+B 拼完 | ... | ... |
| 整体交付前 | ... | ... |

## 里程碑（可选，大项目用）
| 里程碑 | 通过标准 | 状态 |
|-------|---------|------|
| M1: ... | ... | 🔴 |
| M2: ... | ... | 🔴 |

## 风险与待决策项
- 风险 1：...（初步应对策略）
- 待决策 1：[描述决策内容，让用户选择]

## 开发顺序
1. → 模块 B（因为 A 依赖 B）
2. → 模块 A
3. → A+B 集成测试
4. → ...
```

### 原则
- **不写时间估计**（这是给人排期用的，AI 不需要）
- **先测后拼**：有依赖关系的模块，先测通被依赖的，再测依赖它的
- **测试不只是单元测试**：包含 API 测试、集成测试、手动验证步骤
- **每个工作单元写出"预期产出"**，方便后续对照
- **如果有大项目，提前设计里程碑**

---

## 阶段 2：执行开发 + 测试

### 开发节奏

1. **按规划顺序**进行，不跳着开发
2. 每个最小工作单元：
   - 开发 → 产出实际文件/代码
   - 测试 → 写测试或手动验证，记录真实结果
   - 通过 → 打 ✅，记录关键输出（如"API 返回 200，字段正确"）
   - **失败 → 记录错误信息，列出可能原因，重新开发或提决策**

### 真实记录格式

每个单元完成后，在规划文件里更新：

```markdown
### 模块 A - 单元 1
- 状态：✅ / 🔴
- 实际产出：...（文件路径 + 关键内容）
- 测试结果：...（真实输出，非"测试通过"）
- 如失败：原因 + 尝试的修复
```

### 避免幻觉规则
- 写代码前：读相关源文件确认现状
- 改代码后：用 `git diff` 或读文件确认改动真的落盘了
- 测 API：用 curl 或 requests 真实跑一次，记录返回
- 不确定：直接说"不确定 X 的行为，需要实验确认"

### 测试结果判定
- **必须通过**：功能按预期工作（记录实际输出）
- **标记失败**：有报错或输出不符合预期（记录错误信息）
- **跳过需说明**：如果某个测试点暂时跳过，写明原因和依赖条件

---

## 阶段 3：集成点回归测试

### 触发时机
- 两个模块拼完时
- 每次里程碑通过后
- 最终交付前

### 怎么做
1. 列出所有需要回归的测试点（从各模块测试结果里汇总）
2. **逐项执行，记录真实输出**
3. 全部通过 → 标记集成完成
4. 如有失败 → 记录是哪个模块引入的问题，回到阶段 2 修复

### 回归测试报告格式

```markdown
## 集成回归测试 — [时间]

| 测试点 | 结果 | 实际输出/错误 |
|-------|------|-------------|
| 模块 A API | ✅ | 200, {"status": "ok"} |
| 模块 B 写入 | ✅ | 写入成功，ID=123 |
| A→B 调用链 | 🔴 | Error: timeout after 5s |
```

---

## 阶段 4：交付 + 决策清单

### 交付条件
- 所有规划内的开发 + 测试完成
- 集成点回归全部通过
- 或到达里程碑

### 如果中途卡住（缺少信息/需要决策/依赖外部）

**输出决策清单**：

```markdown
## 🚧 暂停 — 等待决策

### 需要你决定的事情

1. **[具体问题]**
   - 选项 A：...（优缺点）
   - 选项 B：...（优缺点）
   - 如果选 A，我会：...

2. **[下一个问题]**（同上格式）

### 当前进度
- 已完成：模块 A（单元 1-3）、模块 B（单元 1）
- 进行中：模块 A（单元 4）— 卡在 [原因]
- 待开始：模块 C、集成测试

### 如果你做了决定
告诉我选项，我继续执行。
```

### 交付物（最终或阶段性）

```markdown
## ✅ 交付 — [日期]

### 已完成
- [功能列表 + 关键输出]

### 测试结果
- 单元测试：X 个，通过 Y 个
- 集成测试：X 个，通过 Y 个
- 回归测试：X 个，通过 Y 个

### 待决策（如果有）
- [同上格式]

### 如何验证
[给用户的验证步骤，让用户确认功能真的work]
```

## 工具使用规范

| 操作 | 工具 | 注意 |
|------|------|------|
| 读源文件 | read_file | 确认现状再改 |
| 写/改代码 | sage_task（附文件路径+内容） | 不自己写 |
| 执行命令 | bash_exec | 记录真实输出 |
| 验证 API | curl | 记录真实响应 |
| 更新规划 | bash_exec 写文件 | 保持规划同步更新 |
| git 状态 | bash_exec | 确认改动已落盘 |

## 使用示例

**用户输入**：
```
帮我开发一个文件批重命名工具，支持按规则重命名，有测试
```

**AI 行为**：
1. 加载 dev-delivery skill
2. 探索项目目录，确认是全新项目
3. 输出开发测试规划（到文件）
4. 用户确认后按规划执行
5. 每个单元完成后更新记录
6. 集成点做回归测试
7. 交付最终结果或决策清单
